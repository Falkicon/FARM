Project Structure¶

The FARM Stack boilerplate follows a monorepo structure, organizing code into clear, logical directories. This guide explains the organization and purpose of each major section.

Directory Overview¶

farm/
├── src/                      # Source code
│   ├── frontend/            # Frontend application
│   │   ├── components/      # Web components
│   │   ├── styles/         # Global styles
│   │   ├── utils/          # Frontend utilities
│   │   └── index.ts        # Frontend entry point
│   ├── backend/            # Backend server
│   │   ├── api/           # API routes
│   │   ├── config/        # Server configuration
│   │   ├── db/            # Database setup
│   │   └── server.ts      # Server entry point
│   └── shared/            # Shared code
│       ├── types/         # TypeScript types
│       ├── constants/     # Shared constants
│       └── utils/         # Shared utilities
├── docs/                  # Documentation
│   ├── api/              # API documentation
│   ├── architecture/     # Architecture docs
│   └── getting-started/  # Setup guides
├── tests/                # Test files
│   ├── frontend/         # Frontend tests
│   ├── backend/          # Backend tests
│   └── e2e/             # End-to-end tests
├── tools/                # Development tools
│   ├── scripts/         # Build scripts
│   └── config/          # Tool configurations
└── public/              # Static assets

Key Directories Explained¶

Source Code (src/)¶

Frontend (src/frontend/)¶

frontend/
├── components/           # Web components
│   ├── app-shell/       # Application shell
│   ├── nav/            # Navigation components
│   ├── forms/          # Form components
│   └── shared/         # Shared components
├── styles/             # Global styles
│   ├── themes/         # Theme definitions
│   └── utilities/      # Style utilities
├── utils/             # Frontend utilities
│   ├── router/        # Routing logic
│   └── state/         # State management
└── index.ts          # Frontend entry point

Backend (src/backend/)¶

backend/
├── api/               # API routes
│   ├── health/       # Health check endpoints
│   ├── auth/         # Authentication endpoints
│   ├── users/        # User management
│   └── files/        # File operations
├── config/           # Server configuration
│   ├── env.ts        # Environment setup
│   └── server.ts     # Server settings
├── db/               # Database
│   ├── prisma/       # Prisma schema
│   └── migrations/   # Database migrations
└── server.ts         # Server entry point

Shared (src/shared/)¶

shared/
├── types/            # TypeScript types
│   ├── models/       # Data models
│   ├── api/         # API interfaces
│   └── utils/       # Utility types
├── constants/        # Shared constants
│   ├── config/      # Configuration
│   ├── api/         # API constants
│   └── ui/          # UI constants
└── utils/           # Shared utilities
    ├── date/        # Date handling
    ├── validation/  # Validation
    └── formatting/  # Formatting

Documentation (docs/)¶

docs/
├── api/              # API documentation
├── architecture/     # Architecture docs
├── getting-started/  # Setup guides
└── assets/          # Documentation assets

Tests (tests/)¶

tests/
├── frontend/         # Frontend tests
│   ├── components/   # Component tests
│   └── utils/       # Utility tests
├── backend/         # Backend tests
│   ├── api/        # API tests
│   └── db/         # Database tests
└── e2e/            # End-to-end tests

Configuration Files¶

  • .env: Environment variables
  • tsconfig.json: TypeScript configuration
  • package.json: Project dependencies and scripts
  • .eslintrc.json: ESLint configuration
  • .prettierrc: Prettier configuration
  • vite.config.ts: Vite configuration
  • typedoc.json: TypeDoc configuration
  • mkdocs.yml: Documentation configuration

Special Files¶

  • README.md: Project overview and quick start
  • CHANGELOG.md: Version history
  • LICENSE: Project license
  • .gitignore: Git ignore patterns
  • .npmrc: npm configuration

  6. Work in src/backend/
  7. Follow API endpoint structure

  8. Update OpenAPI documentation

  12. Maintain backwards compatibility

  14. Write tests alongside source code
  15. Follow testing conventions
  16. Maintain test coverage
